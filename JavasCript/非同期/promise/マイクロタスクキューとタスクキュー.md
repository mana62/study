# タスクキューとマイクロタスクキュー
👉 JavaScript は「同時に2つの処理をしない」
👉 全部 **「順番待ち」**して実行される
👉 **順番待ちの列が2種類ある**

```js
① マクロタスク（setTimeout / イベント など）
② マイクロタスク（Promise.then / catch / finally）
```
👉 優先順位はこう

```js
通常の処理
   ↓
Promise（マイクロタスク） ← 超優先
   ↓
setTimeout / イベント（マクロタスク）
```

```js
[ 同期処理 ] → [ Promise ] → [ setTimeout / click ]
```
👉 Promise は優先される

## setTimeout(0)がすぐに実行されない理由
```js
setTimeout(() => console.log("timeout"), 0);

for(let i=0;i<1e9;i++){}
```
```js
// 結果
for終了
timeout
```
- setTimeout は マクロタスクキュー
- for文は 同期処理
👉 同期処理が終わるまで割り込めない

## Promise が速い理由
```js
setTimeout(() => console.log("timeout"), 0);

Promise.resolve().then(() => console.log("promise"));

console.log("sync");
```

```js
// 結果
sync
promise
timeout
```

```js
同期 → Promise → setTimeout
```
👉 Promise は特別扱いで先に処理される

## なぜ Promise は特別扱いか
👉 仕様でそう決められているから

Promise ：マイクロタスクキュー
setTimeout / イベント：マクロタスクキュー

## JSの実行ルール
下記のループを永遠に繰り返す

① 同期処理をすべて実行
② マイクロタスク（Promise）を全部処理
③ マクロタスクを1個処理
→ ②に戻る

```js
// イメージ
while(true) {
  同期処理
  Promise全部
  setTimeout1個
}
```

```js
setTimeout(() => console.log("A"), 0);

Promise.resolve()
  .then(() => console.log("B"))
  .then(() => console.log("C"));

console.log("D");

// D → B → C → A
```

## なぜこんな仕組み?
Promise は：
- エラー制御
- 連続処理
- 非同期フロー制御
の 根幹 だから setTimeout よりも優先度が高く設計されている

## まとめ
👉 非同期処理はすぐ実行されない
👉 一度キューに入る
👉 Promise は特別優先レーン
👉 setTimeout / イベントは後回し

```js
実行順：

1 同期
2 Promise (.then / catch / finally)
3 setTimeout / イベント
```
