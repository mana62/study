# イベントループで役立つもの

①Web Worker（別スレッド）
②dispatchEvent は同期実行
③DevTools のステップ実行の違い

## ① Web Workerとは？
**JavaScriptを「別スレッド」で動かせる仕組み**

通常のJSは：
- 1本のスレッド
- 1つのイベントループ

つまり：
👉 一度に1つしか処理できない（シングルスレッド）

Web Workerを使うと：
イベントループがもう1つ作られる

```bash
メインスレッド（描画あり）
└ イベントループ

Workerスレッド（描画なし）
└ イベントループ
```
ポイント：
- Worker側には レンダリングは存在しない
- DOMにも触れない
- でも重たい計算はできる

使い所
- 画像処理
- 重いループ
- データ解析
をWorkerにやらせると
👉 メインスレッドは止まらない
👉 画面は固まらない

## ② dispatchEvent
**イベントを手動で発火させる**

通常のイベント（例：click）は：
```bash
イベント発生
↓
タスクキューに入る
↓
後で実行
```
👉 非同期

でも dispatchEvent は：
```js
window.dispatchEvent(new Event("custom"));
```
👉 その場で即実行される
👉 タスクキューに入らない

#### なぜか？
dispatchEvent は
-「イベントを発火させる」ではなく
「イベント処理を今すぐ実行する」
に近い動きだから

## ③ DevTools
**デバッグ・検証・解析**

Chrome DevTools には2種類ある：
🔹 Step over（ステップ）
🔹 Step into / Next function call

### Step（ステップ）
👉 本当のイベントループの動き通りに進む
非同期はちゃんと後で実行される
コードの実行順に忠実

### Step into / Next function call
👉 非同期APIを呼んだ瞬間に
👉 コールバックまでジャンプする

例：
```js
setTimeout(() => console.log(2), 1000);
console.log(1);
```

本来の順番：
```bash
1
（1秒後）
2
```
でも `Next function call` を使うと：
👉 1秒待って
👉 いきなり2の中に入る

つまり：
**デバッグしやすいが、実際の動きとは違う**

## まとめ
### ① Web Worker
- イベントループをもう1つ作れる
- 並列実行できる
- ただしレンダリングはしない

### ② dispatchEvent
- 普通のイベント → 非同期
- dispatchEvent → 同期実行
- タスクキューに入らないのがポイント

### ③ DevToolsのステップ
| 機能                 | 動き             |
| ------------------ | -------------- |
| Step               | 実際のイベントループ通り   |
| Next function call | 非同期をスキップしてジャンプ |

イベントループは：
- メインスレッドに1つ
- Workerごとに1つ
- dispatchEventはキューを使わない
- DevToolsは動きを「疑似的に変えて見せる」ことがある
