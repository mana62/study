# レンダリングがスキップされるタイミング

## レンダリングとは
＝DOMやCSSの状態をもとに、実際に画面に描画すること

具体的に言うと：
- JSがDOMを書き換える
- ブラウザがレイアウトを計算する
- 画面にピクセルとして描画する
👉 この一連の処理が「レンダリング」

```js
document.body.textContent = "1";
document.body.textContent = "2";
document.body.textContent = "3";
```
上記の場合は：
👉 画面には 「3」だけ 表示される

理由：
👉 **JS実行中はレンダリングされないから**

## イベントループの中身
```bash
タスク実行
↓
マイクロタスク実行
↓
レンダリング
↓
次のタスク
```
＝ **JS実行中はレンダリングされない**

つまり：
- JSが動いている間は描画されない
- JSが終わったタイミングで初めて描画される
👉 だから for 文で 10万回回しても最後の値しか見えない

## requestAnimationFrame とは
＝ **「次のレンダリング直前に実行してね」という予約**

```js
requestAnimationFrame(() => {
  console.log("描画直前！");
});
```
＝レンダリング直前に必ず実行される

アニメーション向きなのは：
- ディスプレイの更新タイミング（約60fps）に合わせて実行される
- 無駄なレンダリングが減る

## スキップされる主なタイミング
👉 **「今回は画面更新しなくていいや」とブラウザが判断すること**

### ① ディスプレイの限界（60fps）
モニターは1秒間に約60回しか更新できない

なので：
- イベントループが100回回っても
- 実際に描画されるのは最大60回
- それ以上はスキップ

### ② 画面が見えていないとき
- 別タブ
- 最小化
- バックグラウンド

このときは：
👉 レンダリングも
👉 requestAnimationFrame も
止まる

### ③ 見た目が変わらないとき
DOM変更がなければ：

👉 描画する意味がない
👉 スキップされる

ただし：
**requestAnimationFrame がある場合はスキップされない**

## まとめ
- レンダリングとは「DOMの状態を実際の画面に反映する処理」
- JS実行中は絶対に描画されない
- タスクとタスクの間にだけ描画される
- requestAnimationFrame は描画直前に実行される
- 条件によって描画はスキップされる
